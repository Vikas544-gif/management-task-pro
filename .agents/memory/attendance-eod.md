# Attendance + EOD (Management Task Pro)

## Scoping & roles
- `seesAll = isBoss || isMis` is the key viewer flag on both pages. MIS = `department === "MIS"`.
- **Attendance**: viewers = Boss + Center Head + MIS; `seesAll` gets the center dropdown. Center Head locked to own center (marks its TLs).
- **Who marks what is decided by `canMarkHere` (per selected roster), not a single `canMark` flag**: Center Head → their own center's TLs; Boss → "Center Heads" roster + Head Office (sales centers read-only); **MIS → the "Center Heads" roster only** (as of June 2026 — MIS is no longer fully read-only; it can mark Center Heads' attendance, but stays read-only on every real center). MIS gets `CENTER_HEADS_OPT` in its dropdown (`centerSelectOptions` keys off `isBoss || isMis`). The per-row status cell, Mark-all button, and subtitle all gate on `canMarkHere`.
- **Center-Heads roster drill-down (Boss + MIS)**: clicking a Center Head row expands to that center's Team Leaders with their read-only attendance status for the day (`tlsByHead` by center match, `dayStatus` map over the day's fetched attendance). Drill-down rows are always read-only regardless of marker.
- **Boss can VIEW every center read-only but only MARKS Center Heads + Head Office** (as of June 2026). Boss's `centerSelectOptions` = all `centerOptions` + virtual `CENTER_HEADS_OPT` (defaults to "Center Heads" via `defaultCenter`) — so the Boss can open any sales center to see the TL attendance its Center Head marked, while the sales-center rows render read-only. **Why:** each Center Head marks their own center's TLs; the Boss only marks the people directly under him (Center Heads) + Head Office (no Center Head), but still needs visibility into every center. `perCenterStats` maps real `centerOptions` for the "Present vs Absent by center" chart.
- **"Center Heads" roster (Boss only)**: selecting it makes `members` = all `role === "Center Head"` users so the **Boss can mark the Center Heads' own attendance** (nobody else can — Center Heads are excluded from every real center's roster, so this is the only place they get marked). On mark, store each person's *real* center (`centerFor()`), not the virtual label, so records stay center-accurate. `perCenterStats` (comparison chart) ignores the virtual option (uses real `centerOptions`).
- **EOD server upsert key = `(submittedBy, date)`** (one row per submitter per day) — this is what lets multiple TLs share a center/date and lets a manager edit any team's row by passing that team's userId as `submittedBy`.
- **EOD (per-TL model, edit-for-all as of June 2026)**: each row belongs to a submitter (Team Leaders + the Center Head, since both manage a team). A TL fills their OWN team's card. The **consolidated per-center table is now fully EDITABLE for any viewer who sees it** — Center Head (their center's rows) and Boss/MIS (every center's rows) can inline-edit/fill any team's figures for the **selected single day** (multi-day window stays read-only). Editing keys by the row's userId (`saveForRow` passes `submittedBy: r.id`, the row's center), so a manager doing data-entry updates that team's row, not their own. **Was read-only for Center Head/Boss/MIS before this change.** Same P+A+Attr must equal HC balance rule applies. Grand totals across all centers for Boss/MIS; Center Head sees only their center. Legacy center-level rows ignored (UI aggregates `role === "Team Leader" || "Center Head"` submitters).
- Sidebar gating uses per-item flags `bossOnly/centerHead/mis/tl`; attendance item has `mis:true`, EOD item has `tl:true, mis:true`.
- **EOD server authz (July 2026):** `/api/eod` GET **and POST** are role-scoped server-side. GET: Boss=all, Center Head **and Team Leader**=own center only, MIS/Director=all non-HO, else 403 (TLs previously got 403 — the "center EOD not working" bug; any frontend role added to the EOD page MUST also be added to the server gate). POST mirrors it: Boss=any row, MIS/Director=non-HO, Center Head=own-center rows, TL=own row only (`submittedBy===me.id`), else 403.

## Headcount / shrinkage rule
- **Center members = users in that center with `role !== "Center Head"` AND `role !== "Boss"`** (i.e. the TLs; at Head Office, the staff minus the Boss). The Center Head is excluded in their own center, and the Boss is excluded at Head Office — both are the *marker*, not marked. Applies to both `members` (roster) and `perCenterStats` (Boss comparison) so headcount stays consistent.
- **Why:** attendance is for the team a head manages, not the head/marker; keeps shrinkage comparable across centers. The marker (Center Head per center, Boss at Head Office) is never in their own roster.
- Shrinkage% = `(headcount - (present + 0.5*halfDay)) / headcount * 100`. Half-day counts as 0.5 present; absent + leave count as not-present.

## Persistence
- Upsert by unique keys: attendance `(userId, date)`, eod **`(submittedBy, date)`** (index `eod_user_date_uq`, was `(center, date)`) via Drizzle `onConflictDoUpdate` (idempotent re-fill per TL). `submittedBy` is **notNull** (DB + OpenAPI required) — the new uniqueness/upsert depends on it, and Postgres treats NULLs as distinct so a nullable submitter would silently create duplicates and never upsert. Dates stored as `YYYY-MM-DD` strings; UI computes "today" in IST via `toLocaleDateString("en-CA", { timeZone: "Asia/Kolkata" })`.
- EOD money fields (salesFd, salesMtd, dc) are rupee integers shown as plain en-IN numbers (no ₹ — helper is `fmt()`); "DC" label kept verbatim (Daily Collection). **`hc` (headcount), `present`, and `absent` are all manual integer fields each TL types in** (int notNull default 0, required in OpenAPI) — none are auto-derived from attendance (the EOD page no longer reads attendance data at all). Present/absent are not validated against hc by design. Consolidated table + per-center sum cards + Boss/MIS grand totals all aggregate present/absent alongside the money/hc columns.
- **Shrinkage % is auto-calculated, never stored**: helper `shrinkPct(hc, present) = hc>0 ? max(0, round((hc-present)/hc*1000)/10) : 0` (1 decimal, clamped non-negative, 0 when hc=0). Shown live in the TL card (updates as they type), as a `Shrink%` table column + per-center "Shrinkage" sum card (`shrinkPct(sumHc,sumPresent)`), and a Boss/MIS grand-totals card (`shrinkPct(totalHc,totalPresent)`). Based on hc−present (anyone not present counts), not absent.
- **Attrition** (call-center term for team members who left): manual `attrition` int field (DB notNull default 0, required in OpenAPI), filled per TL alongside present/absent. **Attrition % auto-calculated, never stored**: helper `attrPct(hc, attrition) = hc>0 ? max(0, round((attrition/hc)*1000)/10) : 0`. Mirrors the shrinkage pattern exactly — live display in TL card, `Attr` + `Attr%` table columns, per-center "Attrition" card (shows `count · %`), Boss/MIS grand-totals card. Adding it bumped the consolidated/grand-total grids cols-7→8 and the center table colSpan 8→10. Not validated against hc (attrition>hc → >100% allowed by design).
- API enriches rows with `userName` / `submittedByName`. EOD endpoints are server-scoped (see "EOD server authz" above); attendance remains client-trust.

## Charts
- Attendance page has recharts bars at the bottom: a per-status breakdown (Present/Half Day/Leave/Absent/Unmarked, colored Cells) for the selected center, plus a Boss-only per-center Present vs Absent/Leave comparison. `useListAttendance({date})` returns all centers for the date, so per-center stats are computed client-side with the same TL-only member rule.

## EOD consolidated aggregation (June 2026)
- In the EOD report's consolidated views (per-center `CenterConsolidated` cards + Grand Total tfoot, AND the all-centers summary cards): **Shrinkage and Attrition are AVERAGED across teams**; Sales FD, DC, HC, Present, Absent, **and now Sales MTD are SUMMED across teams** (`sumMtd`/`totalMtd`, label "Total Sales MTD").
- **Sales MTD totaling flipped (June 2026, latest):** previously Sales MTD was per-team ONLY (tiles + Grand Total showed "—"). User later asked "Sales MTD ka total q nhi aa rha" and explicitly wants the center/all combined month-to-date sum, so it is now SUMMED like FD/DC. (Earlier in the same project the user had rejected summing — this is a deliberate later reversal; trust the current code.) When toggling this, keep the page description sentences + the amber range-mode note + the `CenterConsolidated` MTD comment in sync — architect FAILS the build on any stale "not totaled / per-team only / not aggregated" wording.
- **Why:** Sales MTD is a cumulative month-to-date figure each team fills daily; summing across teams gives the center's combined month-to-date sales. Shrinkage/Attrition stay simple averages of per-team % (not a pooled recompute from summed HC/Present/Attrition).
- Averages are over SUBMITTED rows only (`rows.filter(r => r.eod)`) so pending teams don't drag them to 0. Helpers `avgOf`/`round1` at top of Eod.tsx. Attrition shows avg count · avg %. Labels: "Avg Shrinkage / Avg Attrition" (Sales MTD is NOT averaged — do not re-add `avgMtd`/`avgAllMtd`).

## EOD date-range window semantics (June 2026)
- EOD is a single-day report; a multi-day range is a read-only VIEW. Within a window, per team (in `eodByUser`): **everything except Sales MTD is SUMMED across the period** — Sales FD, DC, HC, Present, Absent, Attrition; **Sales MTD is the PEAK (max salesMtd) across the window**. Shrink%/Attr% per row are then derived from the summed HC/Present/Attrition.
- **Why MTD = peak, not latest/sum (within a team's window):** Sales MTD is cumulative month-to-date, grown daily — the month's real figure is its highest point. The latest report can drop to 0 on a reset/closing entry (real data: one TL 10→20→60 then 0 on 06-27); both "latest" AND any other field's "latest" wrongly read the month as 0. NOTE: across TEAMS the per-team MTD peaks ARE now summed into the center/all Grand Total (see "Sales MTD totaling flipped" above) — the peak rule is only for collapsing a single team's multi-day window.
- **Why HC/Present/Absent/Attrition = SUM (not latest):** the user wants the month view to show the period's REAL totals (e.g. total absences, real shrinkage), and a zero/empty closing report must NOT wipe the month. Summing ALL of hc/present/absent together keeps percentages coherent (Present can't exceed HC since both summed); the earlier "summing breaks %" concern only applies if you sum present but keep hc as a snapshot — don't do that. The user iterated to this twice ("ek baar bolne pe nhi hota"): do NOT revert attendance fields to latest-snapshot.
- DB has `uniqueIndex(submittedBy, date)` so single-day collapses each sum to one report; no same-day duplicates possible.
- The "Day-wise present vs absent" chart (per-day trend) and the consolidated totals are now consistent: both sum attendance across the period.

## Absence hides DAILY tasks only (June 2026)
- `isTaskHiddenByAbsence(task, absenceSet)` in iss-task-pro `src/lib/utils.ts` now early-returns unless `String(task.type) === "daily"`. So marking a user absent/leave on a date hides ONLY their DAILY tasks due that day; weekly/monthly/one-time stay visible. Half-day still has no effect. `absenceSet` keys are `userId|YYYY-MM-DD` for status absent||leave (built by `buildAbsenceSet`).
- **Why:** user explicitly wanted absence-hiding limited to daily tasks only (was hiding every task type). Tasks are never deleted/reassigned — pure UI filter shared by TaskList, MyTasks, Dashboard.
