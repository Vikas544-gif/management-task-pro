# Recurring task generation (daily/weekly/monthly "reset")

## Weekend policy (July 2026): daily tasks skip non-working days
DAILY occurrences are NOT generated on the assignee's non-working day: Sunday = everyone off; 1st Saturday of the month = whole company off; other Saturdays = Head Office off EXCEPT the MIS department (centers + MIS work Saturdays). Weekly/monthly/quarterly/annual still generate on schedule (they span the whole period). The rule lives in `isNonWorkingDayFor` (api-server recurrence.ts) and is MIRRORED client-side in iss-task-pro `lib/utils.ts` for the Dashboard off-day banner — keep the two in sync.
**Why:** staff complained daily tasks showed up on days nobody works; occasional Sunday/off-Saturday workers self-serve instead.
**Self-serve:** `POST /tasks/generate-my-daily` (any logged-in user, OWN daily chains only, ignores the skip) backs the Dashboard "Generate Today's Tasks" button shown on off days. These rows are inserted with `autoGenerated=false` so the boot cleanup never deletes them — don't "fix" that flag to true.
**Boot cleanup:** `cleanupNonWorkingDayDailyTasks` (runs every boot) deletes pending+autoGenerated+non-compliance daily occurrences whose lastRunDate falls on the assignee's off day, re-parenting children (newest-first) so chains stay linked. Done tasks are kept (the person actually worked).
**Skip mechanics:** the skip leaves `lastRunDate` untouched, so the next working day generates normally — don't set lastRunDate on skipped days or the next generation is suppressed.

Recurring tasks regenerate by cloning each chain's head into a fresh pending occurrence once per period (period key: daily=YYYY-MM-DD, weekly=Monday YYYY-MM-DD, monthly=YYYY-MM, quarterly=YYYY-Qn, annual=YYYY, all in Asia/Kolkata). History is preserved; old occurrences stay.

## Per-task due-date rules (`tasks.dueRule`)
A recurring task can carry a human due-date rule in the `dueRule` column (e.g. "6th of next month", "Every Wednesday", "31 December"). When set, each generated occurrence's `dueDate` is recomputed for the new period via `computeDueDate({dueDateText: dueRule, frequency: <Capitalized type>}, istToday())` from `@workspace/compliance-dates`, instead of defaulting to the generation day. `dueRule` NULL → falls back to "today" (original behavior for ad-hoc recurring tasks). The same lib parses Daily / weekday / "Nth of month" / "Nth working day" / month-day(annual) rules — it is the single source of truth shared by this generator.
**Why:** lets fixed-deadline duties (compliance-style: TDS by the 6th, GST by the 19th, annual returns on a fixed date) recur with the RIGHT deadline each period rather than all landing on the run date.

## Seeding a recurring task as "already done for its first period"
When bulk-inserting head tasks, set `lastRunDate = periodKey(type, dueDate)` (the period of the seeded occurrence's OWN due date), NOT the current period. The generator only clones when `periodKey(type, now) > lastRunDate`, so seeding with the due-date's period makes the seed serve as that period's occurrence and prevents an immediate duplicate on the next boot/cron. Roll the seed due date forward to the next upcoming occurrence (skip already-passed dates) so the first task isn't born overdue. DAILY seeds must additionally advance to the user's next WORKING day (`isNonWorkingDayFor`) — seed rows are `autoGenerated=false`, so the weekend boot-cleanup will never remove an off-day seed.

## Per-user JD sheet bulk-add = boot seed (prod DB is read-only to the agent)
A person's whole job-description task list (e.g. an Accounts exec's monthly/daily/audit duties per company) ships as an idempotent boot seed in scheduler.ts (advisory-lock, username lookup, opt-out check, exact-title existence check across ALL the user's tasks), NOT direct SQL — dev gets it on restart, prod on republish. Skip sheet rows whose statutory item already exists under another title (e.g. GSTR-1/3B) or you double-generate. `computeDueDate` also understands "Month end" (last day of month) and "Quarter end" (last day of quarter) rule texts for such sheets.

## Rule: the generation job MUST run on server boot, not only via cron
**Why:** node-cron does NOT replay missed runs. The dev server restarts often; if it isn't running at the 00:05 IST cron tick (or restarts after it), that day's occurrences never get created and daily tasks appear "not reset" (heads keep yesterday's last_run_date, no new pending rows).
**How to apply:** `startScheduler()` calls `generateRecurringTasks()` once at startup (with `.catch` logging) in addition to the cron. Keep this boot catch-up — removing it reintroduces the reset bug.

## Safe to run repeatedly
Idempotent via two guards: logical `cur <= head.lastRunDate` check, and DB `onConflictDoNothing` on unique `(parent_task_id, last_run_date)`. Multiple restarts in one period are no-ops after the first generation. Verified: repeated restarts do not create duplicate occurrences.

## Known limitation (not a bug)
Boot catch-up generates only the CURRENT period, not one row per missed historical period during long downtime. Full backfill would be a separate feature.

## Manual trigger
`POST /generate-recurring` runs the same job on demand (useful for testing).

## Idempotency implementation gotcha
The dedupe relies on a PARTIAL unique index over `(parent_task_id, last_run_date)`. In Drizzle's `onConflictDoNothing`, the partial-index predicate goes in `where:` (NOT `targetWhere:`). Getting this wrong silently breaks conflict matching and lets duplicates through.

## Thane weekly templates were wiped & recreated (July 2026)
The one-time boot cleanup "delete every Thane task created before 1 July 2026" (user-requested) also deleted Thane's 4 weekly Team Leader KRA templates, so weekly generation silently stopped for Thane only. Recreated 6 July 2026 by copying Navi Mumbai's weekly templates (4 titles × CH + 4 TLs, assigned_by NULL, auto_generated false) with last_run_date set to the PREVIOUS week key so boot generation immediately cloned the current week. **Lesson:** any bulk task purge by created_at kills recurring chains too — after such a purge, verify every recurring type still has head templates per center. The cleanup filters by created_at < cutoff so the recreated rows are safe from re-deletion. Because prod has a SEPARATE database (read-only to the agent), the fix also ships as an idempotent boot seed `ensureThaneWeeklyTemplates()` in scheduler.ts (advisory-lock guarded; broad user+title existence check on purpose — a live orphan chain plus a new root would double-generate).

## Per-user auto-task opt-out
Some individuals must get NO auto-generated recurring tasks at all (daily/weekly/monthly + the Thane weekly boot seed). The opted-out set = a small CODE baseline `NO_AUTO_TASKS_USER_IDS` in scheduler.ts UNION every user whose DB flag `users.autoTasksEnabled` is false; `optedOutIds(users)` computes it. **Why a hybrid, not pure-DB:** DB overrides proved non-durable in dev — checkpoint/rollback wipes them (same lesson as assign-task-scoping) — so the code baseline is a durable floor, while the DB flag gives Boss/MIS a self-serve toggle from the app (Access Control page) without a code edit each time. ALL generation paths must honor the union: `generateRecurringTasks`, `generateMyDailyTasks` (self-serve button), and `ensureThaneWeeklyTemplates`. Turning the flag OFF also deletes the person's existing tasks: instantly in `PUT /users/:id` (admin-gated field), and again on every boot via `cleanupOptedOutUserTasks()` (also cleans the SEPARATE production DB on first boot after republish; idempotent). Deleting the chain heads is what actually stops regeneration; the opt-out set stops the weekly seed from re-adding templates.
