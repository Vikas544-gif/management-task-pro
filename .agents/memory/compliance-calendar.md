# Compliance Calendar

Accounts-facing statutory compliance tracker in ISS Task Pro (separate from tasks).

## Data model
- `compliance_items` = master list of statutory activities (compliance, activity, dueDateText human string, frequency, `companies` text[], assignedTo, sortOrder, `active`). Now EDITABLE in-app (create/edit/deactivate) by anyone who passes `canAccessCompliance` — the seed is only the initial bootstrap.
- `compliance_status` = per-tick completion, unique on (itemId, company, periodKey). A company-agnostic item (empty `companies[]`) is tracked under the single company key `"ALL"`.
- Seeded idempotently on boot (only when the table is empty) from a transcribed spreadsheet; assignees resolved by name prefix to user ids.
- **Deactivation is soft (`active=false`), never hard delete** — keeps historical `compliance_status` ticks attributable and allows reactivation. The calendar grid hides inactive rows (toggle "Show inactive"); summary counts skip them. POST `/compliance/items` (create, auto-appends sortOrder = max+1), PATCH `/compliance/items/:id` (partial: only keys present in body are changed, so deactivate won't clobber other columns).

## Period & due-date semantics (the subtle part)
- **Period-based, not next-occurrence.** Due date is the deadline that falls *within the current period* (e.g. monthly "Nth of next month" → Nth of the CURRENT month). This is deliberate so an unticked, past-deadline cell shows **overdue** — next-occurrence dates are always in the future and would never surface overdue, defeating the tracker.
- **periodKey must derive from the computed due date, not from `today`.** Otherwise a fixed-date quarterly item (e.g. due 31-Jul) gets bucketed into whatever quarter the calendar is in (Q2 on June 29) instead of the deadline's quarter (Q3), so the tick wouldn't persist across the quarter boundary. Compute due date first, then `periodKeyForDate(frequency, dueDate)`.
- periodKey formats: Daily `YYYY-MM-DD`, Weekly `YYYY-Www`, Monthly `YYYY-MM`, Quarterly `YYYY-Qn`, Annual `YYYY`.

## Access & write integrity
- Read/write gated by `canAccessCompliance`: Boss (Management dept / Boss role) OR all-centers viewer (`isAllCentersViewer`, i.e. MIS/Director) OR Accounts dept. 403 otherwise.
- **Hierarchy (subtree) scoping (server-enforced).** Overseers (`isBoss` OR `isAllCentersViewer` = Boss/Management/MIS/Director) see and can tick ALL items. A non-overseer sees and can tick items whose `assignedTo` is in their org-chart subtree — themselves PLUS all direct/indirect reports — computed with `buildHierarchySet(userId, users)` from `lib/scope.ts`. Enforced on GET (item + status filtering) AND POST /status (403 if outside subtree), not just as a UI filter. **Why:** a manager with reports must see their whole team's items, while an individual contributor sees only their own, and someone with no reports sees none. Started as self-only (`assignedTo === userId`) but corrected to subtree once the reporting structure was confirmed. Unassigned (`assignedTo IS NULL`) items are invisible/immutable to non-overseers by design.
- **Assignment alerts.** Assigning a compliance item to someone (create with `assignedTo`, or reassign to a *new* person) fires the same dual alert as normal tasks: an in-app notification (`notificationsTable`, type `compliance_assigned`) PLUS an email (`sendComplianceAssignmentEmail`, Resend). Skipped when unassigned or self-assigned; PATCH only fires on an actual assignee change (compare prior vs new). Best-effort — wrapped in try/catch so the create/update never fails because email/notif failed. Email always attempted (no `sendEmail` toggle, unlike tasks).
- **Companies are a DB master list, not hardcoded.** The set of company columns is an editable table seeded on boot when empty; new companies are added in-app (inline "Add company" in the item editor) and flow back through the compliance GET response. Names are unique case-insensitively. Grid columns / filter / picker all derive from that list — there is no hardcoded COMPANIES array anymore. **Why:** the group keeps onboarding new entities (e.g. Pixelmint Dubai) and shouldn't need a code change each time. **How to apply:** when adding company-scoped behaviour, read the list from the API, never reintroduce a static array; status cells stay keyed by the company *name* string, so renaming/removing a company would orphan historical statuses (kept intentionally — deactivate, don't delete). Removing a company is the SAME soft pattern: DELETE sets `active=false`; re-adding the same name (case-insensitive) REVIVES the inactive row instead of erroring (active duplicate still 409). GET returns all companies; the grid derives visible columns from `active` ones only. **Edge case:** an item whose every applicable company has been removed has no tickable cell, so `rowDone` treats it as done (true) — returning false would flag it overdue forever.
- POST /compliance/status: `doneBy` comes from `req.user.id` (session), NEVER the request body — body `doneBy` is ignored to prevent attribution spoofing. Server also validates `company` is applicable to the item ("ALL" only for company-agnostic items) and that `periodKey` matches the format regex.
