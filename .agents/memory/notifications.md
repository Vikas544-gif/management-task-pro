# In-app notifications

A `notifications` table + `/notifications` API drive an in-app bell (NotificationBell in the layout header) that polls every 15s, shows an unread badge + dropdown, and pops toasts for newly-arrived unread items.

**When notifications are created (server-side, in routes/tasks.ts):**
- `task_assigned` ‚Üí for the assignee on task create, only when `assignedTo !== assignedBy` (don't notify yourself).
- `task_completed` ‚Üí for the giver (`assignedBy`) on status PATCH, only on the **actual transition** to `done`, and when `assignedBy !== assignedTo`. This ALSO sends an email to the giver (`sendTaskCompletedEmail` ‚Üí `assignedByEmail`), mirroring the assign-email that goes to the assignee. `enrichTask()` exposes `assignedByEmail`; the GET-list bulk-enrich path intentionally omits it (not needed there).

**Why the transition guard + how it's done atomically:** re-saving `{status:"done"}` on an already-done task, or two concurrent "mark done" clicks, would otherwise spam duplicate completion notifications/emails. The PATCH `/:id/status` handler special-cases `status==="done"` with an atomic `UPDATE ... WHERE id=? AND status<>'done' RETURNING` ‚Äî only the single row-winning request notifies+emails; if no row comes back it just returns current state. Do NOT revert to read-prev-then-update (that raced and double-sent).

**No authz by design:** notification endpoints take `userId` from query/body with no ownership check ‚Äî same client-side-only trust model as the rest of this internal tool (see hierarchy-visibility.md). Architect flags this as IDOR every review; do NOT add server authz unless the user explicitly asks for real authentication.

**Toast dedup:** NotificationBell uses a `seenIds` ref initialized to the first-load id set so historical notifications don't toast on mount; only genuinely new unread ids pop a toast.

**Center popup for new assignments:** a brand-new `task_assigned` notification shows a **center modal** (not the corner toast) letting the assignee pick status (Pending=pending / Ongoing=inProgress / Completed=done ‚Äî these are the app's 3 real status values) and add an optional remark. `task_completed` still uses the corner toast.
- Status is saved via the status route (`useUpdateTaskStatus`) so the giver's completion notification still fires; remark is saved separately via `useUpdateTask` (the general PUT does NOT fire notifications).
- Popups are held in a **FIFO queue** (`assignQueue`), not a single slot ‚Äî if several assigns arrive in one poll, or while one popup is open, none are dropped. Critical: the polling effect adds all fresh ids to `seenIds`, so anything not enqueued is lost forever; always enqueue every fresh assign rather than diverting extras elsewhere.
- On save error keep the modal open and show an error; do NOT mark the notification read until the status update succeeds (otherwise silent data loss).

**Desktop (browser) popups ‚Äî per-user opt-in:** Web Notifications API popups fire from NotificationBell's fresh-notification effect (via `showDesktop`, gated through a ref to avoid stale-closure during polling). The on/off toggle is a **per-user** preference keyed `desktopNotif:<userId>` in localStorage. State is shared between the toggle UI and the firing logic via the `use-desktop-notif` hook (a module-level `useSyncExternalStore` store), so toggling re-renders both instantly. The toggle UI lives in the **header (Layout) next to the NotificationBell** (user iterated: not inside the bell dropdown, not in the sidebar ‚Äî wanted it in the top-right header). Enabling requests `Notification.requestPermission()`; denied ‚Üí toggle stays off. Browser must be open for popups (fully-closed ‚Üí only email reminders work).

**Background Web Push (July 2026):** the same popup toggle ALSO manages a real Web Push subscription (service worker `public/sw.js` + `push_subscriptions` table), so popups arrive even with the tab closed (browser must be running; iOS Safari needs PWA install). **Why:** user complained popups only came while the page was open.
- ALL server notification inserts must go through `createNotifications()` (webPush lib) ‚Äî a raw `db.insert(notificationsTable)` silently skips the push. Push send is fire-and-forget, prunes dead (404/410) subscriptions, never fails the calling flow.
- VAPID keys are generated once per environment and persisted in `app_settings` ‚Äî never regenerate/rotate casually; changing keys breaks every existing browser subscription.
- Shared-computer safety: subscribe upserts on endpoint and RE-OWNS the row to the current user; logout awaits `disableBackgroundPush()` BEFORE the logout API call (unsubscribe needs the session cookie). Keep that ordering.
- Users who enabled the toggle pre-push get auto-subscribed via a once-per-page-load effect in `use-desktop-notif` (idempotent).

**Scheduled time-of-day reminders:** tasks have a `dueTime` (text `HH:MM`, alongside `dueDate`). A client-side reminder engine (`use-task-reminders` hook, mounted app-wide from Layout for the logged-in user) polls the current user's own tasks (`useListTasks({assignedTo:userId})`) and a 30s `setInterval` fires a desktop popup once `now >= new Date(`${dueDate}T${dueTime}`)`. It is **restricted to today's date only** (`dueDate === local-today`) so old overdue tasks don't spam popups on every load, and skips `status==="done"`.
- **Dedup must only commit AFTER a successful popup**, never before. The dedup key is `reminded:<userId>:<taskId>:<dueDate>:<dueTime>` in localStorage. `notify()` (on the use-desktop-notif hook) returns `boolean` (false when toggle off / permission not granted / no Notification support); only set the dedup key when it returns true. **Why:** if you set the key before calling notify, a reminder gets permanently marked "sent" while the toggle is OFF, so turning the toggle ON later never shows that task's popup. Architect caught this exact bug.
- `notify()` reads localStorage directly (not the hook's reactive state) to avoid stale-closure inside the interval; the engine holds `notify` and `tasks` in refs so the 30s interval always sees the latest without re-subscribing.
- `dueTime` is threaded through DB schema (`due_time` text col), OpenAPI Task/TaskInput/TaskUpdate, codegen, the AssignTask create form, the TaskList edit modal, and is displayed (‚è∞ HH:MM) next to the date badge in MyTasks and the TaskList table.

**Personal (self-set) reminders ‚Äî separate from assigner dueTime:** users wanted to set their OWN reminder per task, not depend on whoever assigned it. Implemented client-side (no server/DB) in `use-task-reminders` + a üîî button on each My Tasks card opening a modal. Stored per-user in localStorage key `taskReminder:<userId>:<taskId>` (helpers `getPersonalReminder`/`getPersonalReminderConfig`/`setPersonalReminderConfig` exported from the hook). The reminder engine checks BOTH sources: assigner `dueTime` (today-only) and personal reminders (date today-or-future, skips past days).
- **Start/End model (June 2026 redesign):** stored as JSON `{ start, end, afterEndCount, afterEndInterval }`. One popup at `start`, one at `end` (optional), then `afterEndCount` extra "still pending" nags spaced `afterEndInterval` minutes apart after `end`. **History:** the interval was first removed (user found "every how many minutes?" confusing), then the user asked for it BACK once they realised they needed to control the gap ‚Äî so it's now a per-reminder field (default 10 = `AFTER_END_INTERVAL_MIN`, the fallback for legacy values). Reminders stop automatically when the task is marked done (engine `continue`s on `status==="done"`). UI: "Start time" + optional "End time" + a 2-col row "Remind again after end?" (count) / "Every how many minutes?" (interval, disabled until end time set & count>=1).
- **Legacy back-compat:** old `{ at, count, interval }` JSON and old plain-string values are still read ‚Üí mapped to `start`-only (no end, no nags, default interval). Don't drop this read path; localStorage values persist across the redesign.
- **Validation:** `saveReminder` keeps `end` only when strictly `> start` (so `start === end` or `end < start` collapses to start-only); `afterEndCount` forced to 0 when there is no end; `afterEndInterval` floored to >=1.
- **Dedup must key on the occurrence's ABSOLUTE time:** key is `remindedPersonal:<userId>:<taskId>:<occ.getTime()>`. **Why:** keying by occurrence timestamp means editing the reminder later (e.g. adding more after-end nags) never replays an occurrence that already fired at the same time.
- Same after-success-only dedup rule applies (only mark sent when `notify()` returns true), so setting a reminder while the popup toggle is OFF isn't silently lost.
- Reactivity of the üîî indicator/dot in My Tasks is done via a `reminderTick` counter bumped on save/clear + reading localStorage at render (ad hoc but works same-tab); localStorage is per-browser by design, consistent with the desktop-notif toggle.

## Mobile (Expo) push ‚Äî Android FCM setup (July 2026)
- Native APK push needs BOTH halves: `google-services.json` in the iss-mobile artifact root + `android.googleServicesFile` in app.json (client token registration), AND the Firebase service-account key uploaded to Expo as the FCM V1 credential (server delivery via exp.host). Missing either = `getExpoPushTokenAsync` throws (silently caught in lib/push.ts) or Expo push returns InvalidCredentials.
- The FCM V1 key upload was done via Expo's GraphQL API (`api.expo.dev/graphql`, Bearer EXPO_TOKEN): `createGoogleServiceAccountKey` on the account, then `setGoogleServiceAccountKeyForFcmV1` on the androidAppCredentials ‚Äî the eas-cli credentials wizard is interactive-only. The raw service-account JSON was deleted from the repo after upload (secret; lives only in Expo).
- Device tokens live in `expo_push_tokens` (subscribe on login, unsubscribe on logout); server sends through `sendExpoPushToUser` inside `createNotifications()` ‚Äî same "all inserts must use createNotifications" rule as web push.
- **Push tap deep-link (July 2026):** Expo push messages carry `data:{taskId}` (from the notification row) and the mobile root layout listens for taps (response listener + cold-start last-response) ‚Üí routes to `/task/:id`. Dedupe by notification identifier ‚Äî the cold-start "last response" can be the SAME tap the listener already handled, so without the guard the task screen pushes twice. Cold-start check is re-armed per user id so account switches still deep-link.
- **Mobile personal reminders (July 2026):** unlike web's localStorage polling engine, mobile uses OS-level `Notifications.scheduleNotificationAsync` (single date trigger, no end/nag model) with `data:{taskId}` so the existing tap deep-link opens the task. Bookkeeping in AsyncStorage `taskReminders:<userId>` (map taskId‚Üí{notifId, when}). Lifecycle rules: scheduled notifications MUST be cancelled when a task is completed or deleted (else stale reminders fire), hydration must reset state per user (cross-account leakage) and prune past-`when` entries. Web platform hides the bell (expo-notifications can't schedule on web).
- **Expo push MUST be sent one-request-per-token, never batched across a user's tokens:** after the account switch, users had stale tokens from the OLD account's project (`@vikastaskpro/iss-mobile`) alongside new ones; Expo rejects a mixed-project batch entirely with 400 `PUSH_TOO_MANY_EXPERIENCE_IDS`, silently killing delivery to EVERY device. sendExpoPushToUser now sends per-token requests (bad token can't block good ones) and logs the response body on failure. Stale tokens self-prune via DeviceNotRegistered tickets.
- **EAS free plan limit / account switch (July 2026):** old account `vikastaskpro` exhausted its free Android builds (resets Aug 1). User created a NEW Expo account `task-pro-vikas`; EXPO_TOKEN secret now belongs to it, app.json has `owner: task-pro-vikas` + new projectId (eas init created @task-pro-vikas/iss-mobile). **Push caveat:** the FCM V1 service-account key was uploaded only to the OLD account's project and can't be pulled back ‚Äî it must be RE-uploaded (same GraphQL flow) to the new account's androidAppCredentials or push in APKs built on the new account returns InvalidCredentials. DONE July 12 2026: user provided a fresh Firebase service-account JSON, uploaded via the same GraphQL flow to the new account/credentials, raw file deleted from attached_assets. Push works again without a rebuild (credential is server-side).

## Daily-task generation notification (July 2026)
- `generateRecurringTasks` tallies freshly-created DAILY occurrences per user and sends ONE `createNotifications` row per person (type `daily_tasks`, taskId = first new occurrence). **Why:** daily/JD tasks are self-tasks (assignedBy null/self) so the normal assign-notify guard (`assignedTo !== assignedBy`) never fires for them; user wanted a "your daily tasks are ready" push.
- Notification send is gated by the same `claimScheduledJob` once-per-IST-day lock as digests (`daily-task-notify`) ‚Äî generation is idempotent but concurrent boot+cron runs could each create disjoint subsets and double-notify. Released on throw so the day can retry.
- Fires only where generation runs ‚Äî in practice production (published app boot/cron); a server-only change, needs a republish, NOT a new APK.

## Mobile remark editing (July 2026)
- Task detail screen (`app/task/[id].tsx`) has a Remark card: TextField + Save that PUTs `{remark}` only ‚Äî safe because UpdateTaskBody is partial and the server's assignee exception allows status/remark-only PUTs. Draft hydrates once per task id via a ref so poll refetches don't clobber typing.

# All three write paths must emit notifications (POST + PUT + PATCH)
- Tasks have THREE mutating endpoints and each must fire the right notification: POST `/api/tasks` (assign ‚Üí notify assignee), PATCH `/api/tasks/:id/status` (status dropdown / bulk-complete ‚Üí notify giver on done), and **PUT `/api/tasks/:id`** (the full Edit modal + reason-submit + bulk-remark ‚Üí updateTask hook).
- **Bug class:** PUT used to be a dumb update with NO notifications, so completing/reassigning a task via the Edit modal (incl. the TaskList KPI drill-down modal, which opens EditModal) silently skipped notifying the giver / new assignee + completion email. PUT now detects reassignment (assignedTo changed, not self/assigner ‚Üí task_assigned) and completion (status‚Üídone, guarded on prev status, assignedBy‚âÝassignedTo ‚Üí task_completed + email).
- **No double-notify:** status dropdown & bulk-complete use PATCH; reason-submit & bulk-remark use PUT but never set status=done or reassign, so PUT's new branches stay quiet for them.
- **Concurrency:** PUT's done-transition must be atomic like PATCH ‚Äî it runs SELECT‚Ä¶FOR UPDATE + update inside one `db.transaction`, so two concurrent edits flipping to done can't both notify (second blocks, then sees status already done). Plain SELECT-then-UPDATE was the race the first cut shipped with.

## Replay-on-login bug (June 2026)
Bell toast + center assign-popup + the laptop/desktop popup (showDesktop) ALL fire off one shared `fresh` array in NotificationBell's effect (fresh = unread && id not in seenIds). The first-seen `seenIds` snapshot MUST be gated on the notifications query's `isSuccess` (not taken on first render): while loading, `data` is the empty default `[]`, so snapshotting then leaves the set empty and every existing notification looks brand-new on the next render ‚Üí old popups/toasts replay on every login/refresh.
**Why:** user reported already-cleared notifications popping again each login. Fix = `if (!isSuccess) return;` before snapshot + isSuccess in deps. Use isSuccess (not isFetched) so a failed-then-retried fetch doesn't re-trigger the same bug.
The reminder popups in use-task-reminders.ts are a SEPARATE path, deduped permanently via localStorage `reminded:`/`remindedPersonal:` keys ‚Äî never had the replay bug.
