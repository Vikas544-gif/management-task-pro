---
name: Compliance Calendar ↔ All Tasks mirror
description: How Compliance Calendar activities become real auto-generated tasks and stay two-way in sync with the per-company grid
---

Compliance Calendar activities are mirrored into real, editable tasks in All Tasks, one task per active compliance item per period (like recurring tasks).

**Generation:** `generateComplianceTasks()` in api-server scheduler runs on boot + midnight IST cron (alongside `generateRecurringTasks`) and via manual `POST /api/tasks/generate-compliance`. Each task: `type` = the item's **frequency** lowercased (daily/weekly/monthly/quarterly/annual; unknown→one_time) via `COMPLIANCE_FREQ_TO_TASK_TYPE`, so the All Tasks row shows the real cadence instead of "One Time", `category="Compliance"`, `department="Accounts"`, `assignedTo=item.assignedTo`, `assignedBy=null`, `dueDate=computeDueDate`, `complianceItemId` set, `lastRunDate=periodKey`. Idempotent via partial unique index `tasks_compliance_item_period_uq` on (compliance_item_id, last_run_date) WHERE compliance_item_id IS NOT NULL + onConflictDoNothing. Deleting a generated compliance task → it reappears on the next run (expected).

**Recurrence decoupling (critical):** because compliance tasks can now carry type daily/weekly/monthly, `generateRecurringTasks()` MUST skip them — it has `if (head.complianceItemId != null) continue;` as the first loop check. Recurrence for compliance is driven solely by this job + the compliance period key, NOT the parentTaskId daily/weekly/monthly chain. Don't remove that skip or you get double-generation. (Side effects of real types: daily compliance tasks fall under the daily-lock + absence-hiding rules; quarterly/annual aren't in the type tabs/report buckets so they only show under "All".)

**Visibility (Accounts + MIS + Boss only):** `scopeTasks` (scope.ts) routes compliance tasks (`complianceItemId != null`) through `complianceVisible` instead of `baseVisible` — Boss (centerOk) + MIS/Director (all) + Accounts dept (their org-chart subtree); everyone else excluded. This mirrors Compliance Calendar page access. Client `TaskList.tsx` adds `(isMis && t.category === "Compliance")` to its hierarchy filter so MIS (otherwise outer-center-scoped) actually sees them. Existing rows' type backfilled via manual SQL (DEV-only project).

**Applicable companies:** `item.companies` if non-empty, else the single `"ALL"` bucket. Initial task status = done iff every applicable company is already done in `compliance_status` for that period.

**Compliance retired from the product (June 2026), but kept dormant in code.** Decision: the user dropped the whole compliance feature (page, menu, Edit-Task company UI, in-row checkboxes) and wants no auto-generated compliance tasks — they'll create tasks manually instead. So all compliance *UI* and *generation* were removed (scheduler no longer generates; manual generate route gone) and existing compliance tasks were deleted from the DB. **Why kept, not purged:** the backend (`/compliance` routes, `complianceSync`, generation function, compliance tables, `Task.complianceItemId`) was left in place to stay reversible and avoid a risky schema drop. **Invariant that makes this safe:** `syncComplianceStatusFromTask` still runs on every task status update but is a no-op unless a task has a `complianceItemId`; once no task carries one, the dormant sync can't affect anything. If asked to purge fully later, that's a separate destructive pass (drop routes/tables/sync + the schema column).

**Two-way sync** lives in `api-server/src/lib/complianceSync.ts`, both directions write the DB directly (NO HTTP recursion between the tasks and compliance routes):
- Forward (`syncComplianceStatusFromTask`): tasks `PATCH /:id/status` on a compliance-linked task upserts `compliance_status` done/undone for ALL applicable companies of the task's period. Called in every branch of the status handler with `req.user!.id` as actor.
- Reverse (`syncTaskFromComplianceStatus`): compliance `POST /status` recomputes "all applicable companies done" and flips the linked task to done/pending. Called before `res.json`.

**Why:** activity-level task vs per-company grid — marking the task done means the whole activity is done across its companies; the task only auto-completes when the grid shows every applicable company done.

**Shared dates:** date helpers (computeDueDate, periodKeyForDate, urgencyOf, istToday, ymd, etc.) live in lib `@workspace/compliance-dates`; the client's `@/lib/complianceDates` is a re-export shim (still used by TaskList's in-row checkboxes and `complianceDates.test.ts`). Server bundles the lib via esbuild; client resolves it via vite (exports `./src/index.ts`).

**Company master list is public (mirrors category management):** the `complianceCompaniesTable` add/remove endpoints (`GET/POST/DELETE /compliance/companies`) sit ABOVE the compliance role-gate, so any authenticated user can list/add/remove companies from the task-form Company picker (AssignTask/MyTasks/TaskList) — only `requireAuth` applies. **Why:** company add was previously behind the Boss/MIS/Accounts compliance gate → non-privileged users got 403 when adding; user wanted it open like categories. Delete is soft (active=false), and the list is shared with the Accounts Compliance Calendar, so removals affect both surfaces. Client sources the list via `useListComplianceCompanies()` (active-only) and sets `canManageCompanies = true` everywhere.
